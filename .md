# Раздел 2. Технические вопросы

## Kubernetes

### Что таĸое Kubernetes? Для чего его используют?

**Kubernetes** – это инструмент для автоматизации управления контейнерами с программами. 
Он берёт на себя задачи по запуску, масштабированию и мониторингу контейнеров на группе серверов. Если один контейнер выходит из строя, Kubernetes автоматически заменяет его новым. 
Благодаря Kubernetes **приложения работают стабильно** даже при большой нагрузке, а системные администраторы тратят меньше времени на ручное управление.

### Объясните разницу между Pod, Deployment и Service в Kubernetes. Используйте аналогии из реальной жизни, если это поможет.

**Pod** – это базовая единица запуска приложений в Kubernetes. Он содержит один или несколько контейнеров (программ) и работает на сервере (в Kubernetes такой сервер называется узел – это просто компьютер, физический или виртуальный, где запускаются контейнеры). Pod может упасть или перезапуститься, его адрес при этом может измениться.

**Deployment** – это инструмент, который автоматически управляет Pods: создаёт нужное количество копий, обновляет их и следит, чтобы их всегда было достаточно. Если какой‑то Pod перестал работать, Deployment запускает новый на его место. Он помогает поддерживать стабильную работу приложения без ручного вмешательства.

**Service** – это стабильный адрес (IP и имя) для доступа к группе Pods. Он распределяет запросы между доступными Pods и автоматически исключает из работы те, что перестали отвечать. Благодаря Service пользователи и другие программы могут обращаться к приложению по одному и тому же адресу, даже если отдельные Pods перезапускаются или перемещаются на другие серверы.

Аналогия из реальной жизни – **школьное расписание**.
Deployment – система расписания. например, в неделю должны быть 3 урока математики для 5 класса. Deployment создаёт 3 урока математики (Pods): 
* вторник, каб. 101, учитель Иванов И.И.,
* среда, каб. 102, учитель Петров П.П.,
* пятница, каб. 103, учитель Васильев В.В.
  
В школе запущен справочный сервис (Service) с интерактивным расписанием уроков, доступный по постоянной ссылке на сайте школы. Ученик заходит в этот сервис и делает запрос: ```в каком кабинете сегодня математика для 5 класса```. Система выдаёт справку, например, ```Сегодня среда. Математика проходит в каб. 102, учитель Петров П.П.```

Если учитель заболевает (Pod недоступен), Deployment автоматически назначает учителя на замену (новый Pod). Справочная система (Service) обновляет информацию, например, ```Сегодня среда. Математика проходит в каб. 104, учитель Михайлов М.М.```

### Что таĸое Namespace в Kubernetes? Зачем он нужен?

**Namespace (Пространство имён)** – это инструмент для логического разделения ресурсов внутри одного кластера Kubernetes. Он позволяет группировать Pods, Deployments, Services и другие объекты по проектам, командам или этапам разработки (например, среда разработки, тестирования, продакшн). При этом:

   * изменения в одном Namespace не затрагивают другие;
   * доступ к разным Namespace внутри кластера может быть ограничен;
   * для каждого Namespace можно выделить определённый объём ресурсов (CPU, памяти).

## Frontend–разработĸа

### Что таĸое DOM? Каĸ браузер работает с HTML–страницей?

**DOM (Document Object Model, объектная модель документа)** – это представление HTML–документа в виде разветвленной структуры или дерева. Каждый элемент HTML–документа – тег, текст, атрибут или комментарий – отдельный узел DOM–дерева в строгой иерархии.

DOM служит связкой между статичным HTML и динамическими изменениями. Без него браузер отображал бы статичный набор тегов и текста, а страница не была бы интерактивной.

Браузер читает HTML–документ сверху вниз, начиная с глобального объекта ```document```, который представляет всю страницу. Он служит точкой входа для работы с ним через JavaScript с помощью методов добавления и удаления элементов, обработки событий и т.д.

Тег ```html``` становится корневым узлом дерева. Вложенные в него, следующие по иерархии теги ```head``` и ```body``` – дочерними узлами от ```html```, а внутри них – все остальные элементы.

Браузер анализирует CSS‑стили (встроенные или внешние) и создаёт **CSSOM** – объектную модель стилей. Затем он объединяет DOM и CSSOM в **Render Tree** – структуру, описывающую, как должны выглядеть элементы на экране.

Когда DOM построен (даже частично), браузер может выполнять код JavaScript – преобразовывать статичное содержимое в интерактивное, с помощью методов поиска элементов ```document.querySelector('h1')```, изменения текстовых узлов ```element.textContent = 'Новый текст'``` и других.

Изменения в DOM приводят к автоматическому обновлению того, что видит пользователь. Благодаря этому страница становится интерактивной: реагирует на клики, ввод текста и другие действия без перезагрузки.

### В чём разница между HTML, CSS и JavaScript? За что отвечает ĸаждая технология?

**HTML (HyperText Markup Language, язык гипертекстовой разметки)** – технология, которая отвечает за структуру и семантику веб‑страницы с помощью тегов (например, \<h1> для главного заголовка, \<nav> для навигации) и её содержимое.

**CSS (Cascading Style Sheets, каскадные таблицы стилей)** – технология, с помощью которой к элементам HTML применяются различные стили оформления: цвет текста, фон страницы, размеры и отступы элементов и т. п. Ключевое свойство CSS – **каскадность**: стили передаются от родительских элементов к дочерним, что позволяет задавать правила один раз и применять их многократно. CSS также отвечает за адаптивный дизайн – корректное отображение страницы на разных устройствах.

**JavaScript** – язык программирования, который придаёт элементам HTML, оформленным с помощью CSS, интерактивность. Благодаря ему:
* элементы реагируют на действия пользователя (клики, ввод текста, прокрутку и другие);
* содержимое страницы обновляется без перезагрузки;
* реализуется сложная логика сервисов (калькуляторы, чаты, интерактивные карты);
* изменяется структура и стили страницы через DOM в реальном времени.

### Что таĸое React? Почему он популярен?

**React** – это JavaScript‑библиотека для удобной разработки пользовательских интерфейсов (UI). Она основана на компонентном подходе: интерфейс разбивается на независимые компоненты – фрагменты кода, отвечающие за внешний вид и логику работы какого‑либо элемента (например, кнопки, карточки товара, формы ввода).

Такой компонент создаётся один раз, а затем его можно использовать многократно в разных частях приложения. например, кнопка «Подписаться» может менять свой вид (с «Подписаться» на «Вы подписаны») в зависимости от состояния – внутренних данных компонента. При изменении состояния React автоматически обновляет соответствующий участок DOM благодаря механизму виртуального DOM.

React подходит для создания интерактивных веб‑приложений с частыми обновлениями интерфейса, например, чатов или онлайн‑редакторов.

**Причины популярности React**:

* **JSX** – синтаксическое расширение JavaScript для описания структуры компонентов. Оно выглядит как синтез HTML и JavaScript, что обеспечивает лучшую читаемость кода, упрощает его разработку и отладку.
* Повторное использование компонентов – **ускоряет разработку**: создав кнопку или форму один раз, её можно использовать в разных местах без переписывания кода.
* **Виртуальный DOM** – внутренняя копия реального DOM‑дерева. React сначала обновляет виртуальный DOM, затем эффективно вычисляет минимальные необходимые изменения для реального DOM и применяет их. Это значительно повышает производительность при частых обновлениях интерфейса.
* Однонаправленный поток данных – данные передаются сверху вниз от родительских компонентов к дочерним через. Такой подход **делает логику приложения более предсказуемой** и облегчает поиск ошибок.

### Объясните понятие «ĸомпонент» в ĸонтеĸсте frontend–разработĸи. Приведите примеры ĸомпонентов на сайте, ĸоторым пользуетесь ĸаждый день.

**Компонент** – это фрагмент пользовательского интерфейса и его логики, который отвечает за отображение и поведение определённой части веб‑страницы. 
Компоненты могут вкладываться друг в друга: например, страница состоит из крупных компонентов (шапка сайта, секция товаров, подвал), а внутри них находятся мелкие – кнопка «Купить», поле поиска, карточка товара.

Важное свойство компонента – **инкапсуляция**, то есть стили и код одного компонента изолированы от других. Эта упрощает разработку и поддержку веб–страницы. Компонент может также иметь состояние (**state**) – внутренние данные, которые динамически меняют его внешний вид или поведение. например, кнопка «Подписаться» становится «Вы подписаны» после клика пользователя.

Компонент – это автономный блок с HTML‑структурой, CSS‑стилями и JavaScript‑логикой. например, кнопка «Купить» создаётся один раз, а затем многократно используется в разных частях сайта. Собрав библиотеку таких компонентов, разработчики могут быстро собирать интерфейсы, как из деталей конструктора. Результат: меньше дублирования кода, выше скорость разработки.

**Примеры компонентов**:

* поисковая строка веб–страницы;
* плеер на видеоплатформе;
* карусель просмотренных товаров на маркетплейсе;
* кнопки эмодзи под постами в соцсети.

### Что таĸое API? Каĸ frontend взаимодействует с backend?

**API (Application Programming Interface, интерфейс программирования приложений)** – это набор правил и инструментов, который позволяет разным программам обмениваться данными и вызывать функции друг друга.

1. Взаимодействие происходит через HTTP (HyperText Transfer Protocol) запросы к конкретным адресам, эндпоинтам API. HTTP–запрос может состоять из следующих частей:

* метод, например, ```GET``` – получить данные из эндпойнта, ```POST``` – отправить данные;
* адрес эндпойнта;
* заголовки, объявляющие тип данных для передачи, например, ```Content–Type``` – объявляет формат (JSON или XML) данных в теле запроса;
* тело, в котором находятся сами данные.  

2. Backend получает запрос, проверяет его (например, авторизацию), выполняет логику (читает/записывает данные в базу данных) и готовит ответ. Ответ содержит статус – код, показывающий, как backend обработал запрос, например, 200 OK (сервер принял запрос и корректно его обработал). Ответ также содержит заголовки и тело в формате, заданном запросом.

3. JavaScript на frontend’е получает ответ, обрабатывает данные и обновляет DOM – отображает информацию на странице без перезагрузки.

**Взаимодействие frontend и backend** на примере корзины покупок интернет–магазина:

1. Frontend: пользователь нажимает «В корзину» – отправляется ```POST```‑запрос с ID товара на api.site.com/cart.
2. Backend: добавляет товар в корзину пользователя в базе данных и возвращает ответ со статусом ```201 Created```.
3. Frontend: получает подтверждение и обновляет количество товаров в корзине на интерфейсе.

## Unreal Engine

### Что таĸое Unreal Engine? Для ĸаĸих задач его используют?

**Unreal Engine (UE)** – это кроссплатформенный программный комплекс (движок) с набором инструментов для создания интерактивного 3D‑контента. Он используется для разработки видеоигр, архитектурных визуализаций, киноспецэффектов, промышленных симуляций (включая тренажёры) и VR‑программ.

Он сочетает визуальное программирование (**Blueprints**) и работу с **C++**, что делает его доступным как для новичков, так и для профессионалов. Движок поддерживает разработку для ПК, игровых консолей, мобильных устройств и VR/AR‑платформ. 

В распоряжении разработчиков набор технологий, позволяющий собирать проект из готовых решений. например, технология **Lumen** отвечает за освещение сцены. Разработчику не нужно вручную прописывать изменения освещения – технология автоматически адаптирует свет в зависимости от действий игрока или условий сцены.

Открытый исходный код движка доступен для изучения и модификации.

### Что таĸое «слой» (Layer) в Unreal Engine? Для чего нужны слои при разработĸе игры?

**Слой (Layer)** в Unreal Engine – это инструмент группировки и организации элементов игры (объектов, текстур, логики).

Использование слоёв позволяет разработчикам создавать масштабные и детализированные миры, сохраняя контроль над сложностью проекта и эффективностью рабочего процесса.

В зависимости от назначения, бывают **ландшафтные слои** – для текстурирования открытых пространств, **слои материалов** – для сбора элементов из готовых текстур и другие.

**Слои решают несколько задач**:

* **Организация сцен**: сложные уровни делятся на логические группы (ландшафт, персонажи, интерактивные объекты, декор), что упрощает поиск и редактирование.
* **Оптимизация производительности**: тяжёлые объекты (например, детализированная растительность) можно временно скрыть во время разработки.
* **Совместная работа**: дизайнеры могут редактировать ландшафт одного слоя, в то время как программисты настраивают логику игры на другом, без риска случайно изменить чужую работу.
* **Гибкость дизайна**: быстрое тестирование вариантов оформления (например, замена текстуры слоя меняет весь вид локации) без пересоздания объектов.

### Объясните разницу между Actor и Object в Unreal Engine

**Object** – базовый класс элементов движка Unreal Engine. Используется для хранения информации, настроек и ассетов: материалов, текстур, звуков, виджетов UI, таблиц данных и т. д. Объекты этого типа не имеют физической позиции в игровом мире.

**Actor** – подкласс Object, представляющий любой объект, размещаемый в игровом мире (на уровне): персонажа игрока, NPC, источник света, триггер и другие игровые сущности. Actor обладает позицией, вращением и масштабом в 3D‑пространстве, может динамически создаваться и уничтожаться во время игры, а также содержать компоненты для расширения функциональности.

**Пример из игры Need for Speed**:

Actor – это конкретная машина, размещённая на трассе, с координатами, скоростью, состоянием.

Object – это её 3D‑модель, материал корпуса, аудиофайл звука двигателя.

### Что таĸое Blueprint в Unreal Engine? В чём его преимущество перед обычным программированием?

**Blueprint** – система визуального программирования в Unreal Engine на основе графических блоков, соединённых между собой линиями (**нодов**). Позволяет создавать игровую логику без традиционного кодирования.

Каждый нод выполняет определённую функцию, например, воспроизведение звука, получение данных, физика движения объекта. Соединив их вместе, можно **получить игрового персонажа** определённой внешности, поведения, издающего заданные заранее звуки.

Специалист, **не владеющий языками программирования**, например, сценарист или дизайнер, может самостоятельно собрать готового персонажа с помощью библиотеки нодов, и тут же посмотреть, как этот персонаж будет взаимодействовать с игровой средой. Параметры (цвета, звуки, ссылки на объекты) настраиваются прямо в редакторе без правки кода. Результат виден сразу в окне предпросмотра. Это **главное преимущество** Blueprint перед обычным программированием.

Ещё один плюс Blueprint в **наглядности логики** – цепочка выполнения видна визуально, как блоки с функциями и событиями, соединённые линиями связи, как графическое представление алгоритмов.

В Blueprint легко задавать шаблоны и использовать их в дальнейших разработках. 

Однако программирование на С++ всё же необходимо если игра требует **сложной логики или обработки большого объёма данных**, если нужно создавать сложные системы или оптимизировать код.
