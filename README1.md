# Раздел 3. Праĸтичесĸое задание: Доĸументация Kubernetes

## Обзор кластера Kubernetes

Данный раздел содержит общее описание структуры Kubernetes‑кластера, доступного для развёртывания приложений.

### Основная информация о кластере
Кластер Kubernetes предоставляет централизованную среду для запуска и управления контейнеризированными приложениями. 

Вызов осуществляется командой ```kubectl cluster‑info```

**Control Plane (плоскость управления)** — центральный управляющий компонент кластера, обеспечивающий координацию и управление всеми ресурсами. Доступен по адресу: https://cluster.example.com:6443.

**CoreDNS** — сервис для разрешения доменных имён внутри кластера. Он позволяет сервисам находить друг друга по именам вместо IP‑адресов. Работает как часть системного стека Kubernetes.

Для дополнительной диагностики проблем в кластере можно использовать команду:
```kubectl cluster-info dump```

### Пространства имён (namespaces)
В кластере настроены изолированные пространства имён (namespaces), которые позволяют разделить ресурсы по проектам, окружениям или командам. Это помогает избежать конфликтов имён и разграничить доступ.

Примечание: пространства имён ```default```, ```kube‑node‑lease```, ```kube‑public``` и ```kube‑system``` являются системными и предназначены для внутренней работы Kubernetes. Удалять их не рекомендуется — это может привести к неработоспособности всего кластера.

## Инструкция по работе с namespaces

Этот раздел содержит пошаговые инструкции по управлению пространствами имён (namespaces) в Kubernetes. Подробнее о namespaces в [официальной документации Kubernetes](https://kubernetes.io/ru/docs/concepts/overview/working-with-objects/namespaces/#%D0%BF%D1%80%D0%BE%D1%81%D0%BC%D0%BE%D1%82%D1%80-%D0%BF%D1%80%D0%BE%D1%81%D1%82%D1%80%D0%B0%D0%BD%D1%81%D1%82%D0%B2-%D0%B8%D0%BC%D1%91%D0%BD).

### Как просмотреть существующие namespaces

Чтобы увидеть список всех доступных пространств имён в кластере, выполните команду:

```kubectl get namespaces```

Результат выполнения команды:

|NAME|STATUS|AGE|
|-|-|-|
|default|Active|365d|
|kube-node-lease|Active|365d|
|kube-public|Active|365d|
|kube-system|Active|365d|
|development|Active|180d|
|staging|Active|180d|
|production|Active|180d|
               
Пояснение к столбцам:

**NAME** — название пространства имён;

**STATUS** — статус (обычно Active для работающих namespace);

**AGE** — время существования namespace.

### Как создать новое пространство имён

Для создания нового namespace используйте команду:

```kubectl create namespace NAMESPACE_NAME```

Пример: создание namespace с именем test-env:

```kubectl create namespace test-env```

Ожидаемый результат:

namespace/test-env created

### Как переключиться между namespaces

В Kubernetes нет «глобального» переключения контекста на namespace для всех последующих команд. Вместо этого нужно явно указывать namespace в каждой команде с помощью флага ```-n``` или ```--namespace```.

**Способ 1. Указание namespace в команде**

Добавьте флаг ```-n NAMESPACE_NAME``` к любой команде ```kubectl```, чтобы выполнить её в нужном пространстве имён.

Примеры:

Просмотр подов в namespace ```development```:

```kubectl get pods -n development```

Просмотр деплойментов в namespace ```staging```:

```kubectl get deployments -n staging```

**Способ 2. Настройка контекста**

Если вы планируете долго работать в одном namespace, можно настроить контекст с помощью ```kubectl config```. Это позволит не указывать флаг ```-n``` в каждой команде.

Создать новый контекст или изменить текущий:

```kubectl config set-context --current --namespace=NAMESPACE_NAME```

Пример: переключение на namespace ```production```:

```kubectl config set-context --current --namespace=production```

Теперь все последующие команды ```kubectl``` будут выполняться в namespace ```production``` по умолчанию.

**Проверка текущего namespace контекста**

```kubectl config view --minify | grep namespace```

Если namespace не задан, команда не вернёт результата — значит, используется ```default```.

**Важные замечания:**

**Права доступа.** Для создания новых namespace и работы с ними могут **потребоваться соответствующие права доступа** (RBAC). Обратитесь к администратору кластера, если у вас нет прав на создание namespace.

**Системные namespace.** Не изменяйте и не удаляйте системные пространства имён (```kube‑system```, ```kube‑public``` и т. д.), так как это может **нарушить работу кластера**.

**Очистка.** Если namespace больше не нужен, его можно удалить командой:

```kubectl delete namespace NAMESPACE_NAME```

Удаление namespace приведёт к **удалению всех ресурсов** внутри него.

## Описание деплойментов

Этот раздел содержит информацию о запущенных приложениях в кластере (в пространстве имён ```development```) и объясняет, как интерпретировать их статус. (Подробнее о деплойментах в [официальной документации Kubernetes](https://kubernetes.io/docs/concepts/workloads/controllers/deployment/) англ. версия)

### Как получить список развёрнутых деплойментов

Чтобы просмотреть список деплойментов в конкретном namespace, используйте команду:

```kubectl get deployments -n NAMESPACE_NAME```

Пример для namespace ```development```:

```kubectl get deployments -n development```

**Результат выполнения команды:**

|NAME|READY|UP‑TO‑DATE|AVAILABLE|AGE|
|-|-|-|-|-|
|auth-service|2/2|2|2|45d|
|api-gateway|3/3|3|3|90d|
|user-service|2/2|2|2|60d|
|payment-service|1/1|1|1|30d|
|notification-service|1/1|1|1|15d|

**Пояснение к столбцам в выводе** ```kubectl get deployments```

* ```NAME``` — имя деплоймента (приложения).
* ```READY``` — показывает соотношение текущих работающих реплик к желаемому количеству. Формат: ```<текущие>/<желаемые>```. Например, ```2/2``` означает, что запущены и работают обе ожидаемые реплики.
* ```UP-TO-DATE``` — количество реплик, обновлённых до последней версии конфигурации (декларативного описания приложения).
* ```AVAILABLE``` — количество реплик, которые готовы принимать трафик (успешно прошли проверки готовности — *readiness probes*).
* ```AGE``` — время с момента создания или последнего масштабного изменения деплоймента.

**Что означает статус деплоймента**

Статус деплоймента считается **здоровым** (всё работает корректно), если:

* значения в столбцах ```READY```, ```UP-TO-DATE``` и ```AVAILABLE``` совпадают;
* количество доступных реплик (```AVAILABLE```) соответствует желаемому числу экземпляров приложения.

**Примеры интерпретации:**

* ```2/2 2 2``` — деплоймент полностью готов: запущены 2 реплики, обе обновлены и доступны для обработки запросов.
* ```1/2 1 1``` — проблема: ожидается 2 реплики, но запущена только 1. Приложение может работать нестабильно или с ограниченной производительностью.
* ```0/2 0 0``` — все реплики недоступны. Приложение не работает.

## Инструĸция по проверĸе статуса подов

Этот раздел содержит инструкции по просмотру подов (Pods) в кластере и анализу их состояния. Подробнее о деплойментах в [официальной документации Kubernetes](https://kubernetes.io/ru/docs/concepts/workloads/pods/)

### Как посмотреть список подов в namespace

Чтобы вывести список всех подов в определённом пространстве имён, используйте команду:

```kubectl get pods -n NAMESPACE_NAME```

Пример для namespace ```development```:

```kubectl get pods -n development```

**Результат выполнения команды:**

|NAME|READY|STATUS|RESTARTS|AGE|
|-|-|-|-|-|
|auth-service-5f8d9c7b4d-x7k2m|1/1|Running|0|2d|
|auth-service-5f8d9c7b4d-p9n3q|1/1|Running|0|2d|
|api-gateway-6c7b8d9f5c-m4k8p|1/1|Running|0|5d|
|api-gateway-6c7b8d9f5c-n2w7r|1/1|Running|0|5d|
|api-gateway-6c7b8d9f5c-q1t5s|1/1|Running|0|5d|
|user-service-7d8e9f0a6d-k3j6h|1/1|Running|0|1d|
|user-service-7d8e9f0a6d-l5m9n|1/1|Running|0|1d|
|payment-service-8e9f0a1b7e-o2p4q|1/1|Running|0|12h|
|notification-service-9f0a1b2c8f-r3s5t|1/1|Running|0|6h|

**Пояснение к столбцам в выводе** ```kubectl get pods```

* ```NAME``` — уникальное имя пода, включающее имя деплоймента и идентификатор реплики.
* ```READY``` — показывает соотношение запущенных контейнеров к общему числу контейнеров в поде. Формат: ```<запущенные>/<всего>```. Например, ```1/1``` означает, что 1 контейнер из 1 запущен и готов к работе.
* ```STATUS``` — текущий статус пода. Основные статусы:
	* ```Running``` — под запущен и работает;
	* ```Pending``` — под создан, но ещё не размещён на ноде или загружается образ контейнера;
	* ```ContainerCreating``` — идёт создание контейнеров в поде;
	* ```Error``` — в поде произошла ошибка;
	* ```CrashLoopBackOff``` — контейнер постоянно перезапускается из‑за ошибок;
	* ```Completed``` — под выполнил свою задачу (для одноразовых задач);
	* ```Terminating``` — идёт удаление пода.
* ```RESTARTS``` — количество перезапусков контейнеров в поде. Большое число может указывать на проблемы с приложением.
* ```AGE``` — время с момента создания пода.

### Как получить подробную информацию о поде

Для детальной диагностики используйте команду ```kubectl describe```, чтобы получить полную информацию о поде, включая события, состояние контейнеров и смонтированные тома:

```kubectl describe POD_NAME -n NAMESPACE_NAME```

Пример для namespace ```development```:

```kubectl describe pod auth-service-5f8d9c7b4d-x7k2m -n development```

В выводе команды обратите внимание на разделы:

* Containers — состояние каждого контейнера в поде;
* Conditions — текущее состояние пода (готовность, живучесть);
* Events — хронологический список событий, связанных с подом (может помочь в диагностике проблем).

### Дополнительные команды для мониторинга

**Просмотр подов во всех namespace:**

```kubectl get pods --all-namespaces```

**Расширенный вывод с IP‑адресами и нодами:**

```kubectl get pods -n NAME -o wide```

**Фильтрация подов по статусу (пример для статуса ```Running```):**

```kubectl get pods -n NAME --field-selector=status.phase=Running```

## Инструĸция по просмотру логов

Этот раздел содержит инструкции по получению и анализу логов приложений, запущенных в Kubernetes. 

### Как посмотреть логи пода

Чтобы вывести логи конкретного пода, используйте команду:

```kubectl logs POD_NAME -n NAMESPACE_NAME```

Пример просмотра логов пода ```auth-service-5f8d9c7b4d-x7k2m``` в namespace ```development```:

```kubectl logs auth-service-5f8d9c7b4d-x7k2m -n development```

### Как посмотреть логи пода с несколькими контейнерами

Если в поде несколько контейнеров, необходимо явно указать имя контейнера с помощью флага ```-c```:

```kubectl logs POD_NAME -c CONTAINER_NAME -n NAMESPACE_NAME```

Пример:

```kubectl logs api-gateway-6c7b8d9f5c-m4k8p -c api-container -n development```

Чтобы узнать **имена контейнеров** в поде, используйте ```kubectl describe```:

```kubectl describe pod POD_NAME -n NAMESPACE_NAME```

### Полезные флаги для команды ```kubectl logs```:

* ```--tail=N``` — показать только последние ```N``` строк лога для быстрого просмотра без прокрутки всего вывода.

```kubectl logs POD_NAME -n NAMESPACE_NAME --tail=N```

* ```-f``` или ```--follow``` — следить за логами в реальном времени при отладке или мониторинге работы приложения.

```kubectl logs -f POD_NAME -n NAMESPACE_NAME```

* ```--since=X``` — показать логи за последние ```X``` времени (например, 1h — за последний час, 10m — за последние 10 минут).

```kubectl logs POD_NAME -n NAMESPACE_NAME --since=X```

* ```grep``` — показать логи по ключевому слову, например "ERROR" с помощью утилиты Unix-подобной системы ```grep```.

```kubectl logs POD_NAME -n NAMESPACE_NAME | grep "ERROR"```

Другие полезные флаги для команды ```kubectl logs``` можно посмотреть в [официальной документации Kubernetes](https://kubernetes.io/docs/reference/kubectl/generated/kubectl_logs/) англ. версия.

### Как посмотреть логи всех подов деплоймента

Для просмотра логов всех реплик деплоймента можно использовать метки (```labels```). Сначала найдите метку деплоймента:

```kubectl get deployments DEPLOYMENT_NAME -n NAMESPACE_NAME --show-labels```

Затем получите логи всех подов с этой меткой:

```kubectl logs -l LABEL_KEY=LABEL_VALUE -n NAMESPACE_NAME```

## Частые проблемы и их решения

Раздел содержит типичные проблемы, которые могут возникнуть при работе с кластером Kubernetes, и пошаговые инструкции по их устранению.

### Не удаётся подключиться к кластеру (```Unable to connect to the server```)

**Когда возникает**:

* при выполнении любой команды ```kubectl```;
* если изменился kubeconfig или истёк срок действия токена;
* при проблемах с сетью.

Сообщение об ошибке выглядит следующим образом:

```The connection to the server URL was refused — did you specify the right host or port? Unable to connect to the server: net/http: TLS handshake timeout```

**Решение**:

1. Проверьте доступность API‑сервера: ```ping cluster.example.com```.
2. Убедитесь, что kubeconfig настроен корректно: ```kubectl config current-context```.
3. Проверьте файл kubeconfig: ```kubectl config view```.
4. Если используете токен, обновите его.
5. Проверьте сетевое подключение и настройки прокси.

### Деплоймент не масштабируется (```Desired replicas not matching```)

**Когда возникает**:

* в ```READY``` меньше реплик, чем в ```AVAILABLE```;
* проблемы с ресурсами или конфигурацией.

Пример при исполнении команды ```kubectl get deployments```:

```2/3 2 2``` — проблема: ожидается 3 реплики, но запущены только 2.

**Решение**:

1. Найдите поды с проблемами: ```kubectl get pods -n NAMESPACE_NAME --field-selector=status.phase!=Running```.
2. Изучите логи проблемных подов: ```kubectl logs POD_NAME -n NAMESPACE_NAME```.
3. Проверьте ресурсы (например, загрузку CPU/памяти): ```kubectl top nodes``` ```kubectl top pods -n NAMESPACE_NAME```.
4. Проанализируйте конфигурацию деплоймента: ```kubectl describe deployment DEPLOYMENT_NAME -n NAMESPACE_NAME```.

### Ошибка создания namespace (```AlreadyExists```)

**Когда возникает**:

* если используется неуникальное имя в кластере.

Сообщение об ошибке выглядит следующим образом:

```Error from server (AlreadyExists): namespaces "development" already exists```

**Решение**:

1. Просмотрите существующие namespace: ```kubectl get namespaces```.
2. Используйте другое имя для нового namespace.
3. Либо используйте существующий namespace.
